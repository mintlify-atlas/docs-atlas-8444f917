---
title: "Stable ID System: Content-Based Hashing in Memory Pak"
description: "How Memory Pak generates content-based IDs for games, LEGO Dimensions figures, and Skylanders so that saved states survive database updates."
---

When Memory Pak ships a database update — new games added, entries reordered, release dates corrected — every item must keep the same ID it had before. If IDs shifted, the `owned`, `wishlist`, and `notes` data stored by the user would silently detach from the wrong items. Memory Pak solves this by deriving IDs from item content rather than from list positions or auto-incremented integers.

## Game IDs

Game IDs are produced by `generate_stable_id` in `src/game_data.rs`:

```rust
fn generate_stable_id(console_id: &str, title: &str) -> String {
    let mut hasher = DefaultHasher::new();
    console_id.hash(&mut hasher);
    title.hash(&mut hasher);
    format!("{}-{:x}", console_id, hasher.finish())
}
```

The two inputs — `console_id` and `title` — are hashed together using Rust's `std::collections::hash_map::DefaultHasher`. The result is formatted as a lowercase hex string and prefixed with the console ID.

**Format:** `"{console_id}-{16 hex digits}"`

**Example:** `"nes-a1b2c3d4e5f67890"`

<Note>
`DefaultHasher` is not cryptographically secure, but cryptographic strength is not required here. The goal is stable, collision-resistant output for a fixed (console_id, title) pair — `DefaultHasher` provides this within a single compiled binary.
</Note>

### Filename-to-console-id mapping

The JSON filenames in `database/` do not always match the console IDs used in game IDs. `map_filename_to_console_id` in `game_data.rs` performs the translation at load time:

| JSON filename | Console ID used in hash |
|---|---|
| `gameboy` | `gb` |
| `gameboycolor` | `gbc` |
| `gameboyadvance` | `gba` |
| `nintendods` | `ds` |
| `nintendo3ds` | `3ds` |
| `playstation` | `ps1` |
| `playstation2` | `ps2` |
| `playstation3` | `ps3` |
| Everything else | filename as-is |

### Extracting console ID from a game ID

Because the console ID is the prefix before the first `-`, it can be recovered without storing it separately:

```rust
pub fn get_console_from_id(game_id: &str) -> &str {
    game_id.split('-').next().unwrap_or("")
}
```

This works correctly for all console IDs in the database. For IDs like `3ds`, `ps1`, or `xbox360`, `split('-').next()` returns the full prefix before the hash segment.

<Warning>
Many console IDs in `consoles.json` contain dashes (for example `nes-toploader`, `switch-lite`). However, none of these variant consoles have a game database file, so their IDs are never used as the prefix in a game ID. `get_console_from_id` is only applied to IDs returned by `generate_stable_id`, which uses the console IDs from `map_filename_to_console_id` — all of which are dash-free. If you add a new game database file for a console whose ID contains a dash, `get_console_from_id` will return only the portion before the first dash, producing an incorrect result.
</Warning>

## LEGO Dimensions figure IDs

Figure IDs are generated by `figure_id` in `src/lego_dimensions.rs`:

```rust
pub fn figure_id(figure: &LegoDimensionFigure) -> String {
    fn slugify(value: &str) -> String {
        let slug = value
            .to_lowercase()
            .chars()
            .map(|c| if c.is_ascii_alphanumeric() { c } else { '-' })
            .collect::<String>();

        slug.split('-')
            .filter(|segment| !segment.is_empty())
            .collect::<Vec<_>>()
            .join("-")
    }

    format!("lego-{}-{}", slugify(&figure.pack_id), slugify(&figure.name))
}
```

**Format:** `"lego-{pack_id_slug}-{name_slug}"`

**Example:** `"lego-71174-batman"`

The `slugify` helper lowercases the input, replaces every non-ASCII-alphanumeric character with `-`, then collapses consecutive dashes by splitting and re-joining non-empty segments.

## Skylanders IDs

Skylander IDs are generated by `skylander_id` in `src/skylanders.rs` using the same `slugify` logic:

```rust
pub fn skylander_id(skylander: &Skylander) -> String {
    format!(
        "skylander-{}-{}-{}",
        slugify(&skylander.game),
        slugify(&skylander.name),
        slugify(&skylander.category)
    )
}
```

**Format:** `"skylander-{game_slug}-{name_slug}-{category_slug}"`

**Example:** `"skylander-spyro-s-adventure-spyro-core-skylanders"`

The game `"Spyro's Adventure"` slugifies to `"spyro-s-adventure"` (the apostrophe becomes a dash, consecutive dashes are collapsed). Including both `category` and `game` as segments allows the same character name to coexist across different Skylanders titles and release waves without collision.

## Stability guarantees

<AccordionGroup>
  <Accordion title="Safe database operations">
    The following changes to the database JSON files will not affect any existing IDs and are safe to ship:

    - Adding new games, figures, or Skylanders entries
    - Reordering items within a file
    - Correcting publisher names, release dates, or other non-key fields
    - Adding new JSON files for new consoles
  </Accordion>
  <Accordion title="Breaking database operations">
    The following changes will produce a different ID for the affected item. Any user state tied to the old ID will become orphaned:

    - Renaming a game title (game IDs hash the title)
    - Changing a `console_id` (game IDs hash the console ID)
    - Renaming a `pack_id` or figure `name` in `legodimensions.json`
    - Renaming a Skylander's `game`, `name`, or `category`
  </Accordion>
  <Accordion title="Export format compatibility">
    The `ExportData` struct uses `#[serde(default)]` on the LEGO Dimensions and Skylanders state vectors:

    ```rust
    pub struct ExportData {
        pub version: String,
        pub export_date: String,
        pub console_states: Vec<ConsoleState>,
        pub consoles: Vec<ConsoleExportData>,
        #[serde(default)]
        pub lego_dimensions_states: Vec<LegoDimensionState>,
        #[serde(default)]
        pub skylanders_states: Vec<SkylanderState>,
    }
    ```

    This means export files created before LEGO Dimensions and Skylanders support was added can still be imported successfully — the missing fields default to empty vectors rather than failing deserialization.
  </Accordion>
</AccordionGroup>

<Tip>
If you must rename a title or console ID in the database, generate the new ID for each affected item and write a one-time migration that rewrites the stored state files, remapping old IDs to new ones before shipping the change.
</Tip>
