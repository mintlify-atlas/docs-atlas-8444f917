Memory Pak tracks consoles and hardware variants across seven manufacturers. Each console has a unique ID used throughout the app â€” in game database filenames, export files, and saved states. The full list is defined in `database/consoles.json`, which is embedded into the binary at compile time via `include_str!`.

## Console struct

Each entry in `consoles.json` deserializes into the following Rust struct defined in `src/main.rs`:

<ResponseField name="id" type="string" required>
  Short unique identifier for the console. Used as the key for console states, game IDs, and database filenames. Example: `"nes"`, `"ps2"`, `"switch"`.
</ResponseField>

<ResponseField name="name" type="string" required>
  Full display name shown in the UI. Example: `"Nintendo Entertainment System"`.
</ResponseField>

<ResponseField name="manufacturer" type="string" required>
  Console manufacturer. One of: `Nintendo`, `Sega`, `Sony`, `Microsoft`, `Atari`, `Meta`, `Oculus`, `NEC`, `SNK`, `Valve`, `Bandai`.
</ResponseField>

<ResponseField name="year" type="number" required>
  Year of release for this hardware revision.
</ResponseField>

<ResponseField name="variant" type="string">
  Optional hardware variant label. Defaults to `null` when not specified (uses `#[serde(default)]`).
</ResponseField>

## Console IDs and game database filenames

Console IDs are the canonical identifiers used throughout Memory Pak. When a game database file exists for a console, its filename in `database/` may differ from the console ID. See the [game database reference](/reference/game-database#filename-to-console-id-mapping) for the full mapping table.

Not every console in this list has a corresponding game database file. Consoles without a game database file (for example, `ps4`, `xboxone`, `switch`, and all hardware variants) can still be tracked as owned, favorited, or wishlisted â€” they simply have no game list to browse.

<Note>
  Console IDs used in [export files](/reference/export-format) must exactly match the `id` values in this list. An export file that references an unknown console ID will import without error â€” the unknown states are merged into the app's state map and will appear if a matching console is added in a future release.
</Note>
